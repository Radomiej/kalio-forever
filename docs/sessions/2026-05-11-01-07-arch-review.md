# Architecture Review

## What was reviewed

- Scope: backend modules, frontend features/stores, shared types, audit pipeline, coverage, dead code, duplicate tests, security-sensitive paths.
- Focus: whether the current architecture is simpler and stable enough, where dead code and duplicate maintenance exist, and where repo rules diverge from implementation.
- Date: 2026-05-11 01:07 local time.

## Key findings

- Module isolation: RED — documented module boundaries are not the real architecture; multiple backend modules import sibling services directly and `ToolModule` acts as a broad integration hub.
- Type contracts: YELLOW — shared contracts remain centralized, but `packages/@kalio/types/src/index.ts` has become a large God file.
- Scalability: YELLOW — backend coverage is strong and the max-iterations limit is configurable, but the system still relies on SQLite single-writer behavior, local VFS growth, and a single in-process orchestration path.
- Security: RED — LLM and image secrets are encrypted at rest, but embedding credentials are still stored plaintext; RA-App iframe messaging checks `event.source` but not origin/CSP hardening.
- Frontend state coherence: YELLOW — reconnect/session re-identify and chunk cleanup exist, but the state model is spread across large stores/components and remains expensive to reason about.
- Audit trustworthiness: RED — the audit pipeline underreports real issues; commented silent catches and nested knip unused-file results are missed by the generated aggregate report.
- Dead code / duplicates: YELLOW — runtime dead code is moderate, but there are orphaned/stub files and overlapping `.spec`/`.test` suites that duplicate maintenance effort.

## Critical items

- Encrypt embedding credentials at rest to match LLM and image credential handling.
- Stop treating the generated aggregate audit report as authoritative until silent-catch and knip file-report parsing are fixed.
- Break the `tool-registry.service.ts` ↔ `subagent.tool.ts` cycle and remove the `ModuleRef` service-locator fallback.
- Decide whether backend “modules” are real boundaries or just folders; either enforce ports/contracts or collapse them into an explicit integration layer.

## Open questions

- Should `ToolModule` remain a single integration point, or should tools be registered per domain and composed one level higher?
- Do you want to preserve the current “module” naming even if service-level imports between modules stay allowed?
- Should unused UI files be deleted now, or are some of them placeholders for in-progress screen rewrites?
- Is the audit pipeline intended to gate CI, or is it currently advisory only?

## Next steps

- Fix audit false negatives first: broaden silent-catch detection and parse knip nested `issue.files` entries.
- Encrypt embedding credentials and add parity tests with `CredentialsService` / `ImageConfigService`.
- Extract a `ToolCatalogPort` or equivalent to remove the subagent/tool registry cycle.
- Prune orphaned UI/stub files and merge overlapping `.spec`/`.test` suites into one naming convention.
- Split the largest files only after deciding the real ownership boundaries, otherwise the refactor will just spread the same coupling across more files.
