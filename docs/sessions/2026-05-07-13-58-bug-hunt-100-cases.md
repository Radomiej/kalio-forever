## Key decisions

- Did not fix the discovered bugs after confirming them; the user explicitly wanted bug hunting and reporting, not implementation repair.
- Preferred table-driven specs in existing test files or small new spec files to maximize unique fail-case coverage per edit.
- Avoided widening into heavy E2E/integration once the lightweight unit/component surfaces were clearly producing real failures.

## Open questions

- Several failures now overlap with product/API contract questions, not just missing guards:
  - should invalid frontend numeric input revert to the last valid value or clamp to backend defaults?
  - should MCP manual args support shell-style quoted tokenization or remain explicit/simple?
  - should tool runtime validation return normalized domain errors (`INVALID_*`) or typed Nest exceptions?
- `chat.service.ts` still has known shared-type drift around `EMPTY_ASSISTANT_RETRY_EXHAUSTED`, but that was not needed to cross the 100-case threshold.
